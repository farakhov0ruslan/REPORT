# Отчёт по проекту ATLAS

---

## Суть приложения

**ATLAS** — персональный планировщик городских путешествий. Проблема, которую решает приложение: самостоятельное планирование поездки требует большого времени — нужно найти достопримечательности, оценить их стоимость и длительность, распределить по дням и не перегрузить расписание. Туристы либо платят за организованный тур, либо тратят часы на составление маршрута вручную.

ATLAS автоматизирует этот процесс. Пользователь проходит пятишаговый опрос (интересы, стиль поездки, бюджет, темп, количество дней), на основе которого система формирует персональную подборку мест. Затем пользователь добавляет понравившиеся места в «подборку» (корзину) и нажимает одну кнопку — алгоритм сам распределяет места по дням с учётом времени посещения и бюджета, расставляет временны́е метки и считает итоговую стоимость. Готовый маршрут можно отредактировать, добавить заметки, скачать в `.txt` или распечатать.

Приложение реализовано как полностью клиентское — без бэкенда, базы данных и реальных внешних API. Все данные о местах и логика рекомендаций работают через моки, что делает проект полностью самодостаточным для демонстрации.

---

## Предметная область

Городской туризм — планирование короткой поездки (1–7 дней) по российскому городу. Пользователь имеет конкретные интересы (музеи, еда, музыка, шопинг и т.д.), ограниченный бюджет и время. Приложение помогает за несколько шагов перейти от «хочу куда-то поехать» к конкретному расписанию с адресами, временем посещения и стоимостью.

## Как опрос влияет на подборку мест

После завершения онбординга приложение отправляет предпочтения на `POST /api/recommendations`. Алгоритм `getRecommendations` применяет их последовательно:

**1. Интересы → жёсткий фильтр по категории**

Из всех мест остаются только те, чья категория выбрана пользователем.

```
Выбрал: Музеи + Еда
→ Исключаются: Музыка, Экскурсии, Шопинг, Экстрим
```

**2. Бюджет → жёсткий фильтр по цене**

| Выбор | Лимит цены |
|---|---|
| Экономно | до 500 ₽ |
| Комфорт | до 1 500 ₽ |
| Без ограничений | любая цена |

Бесплатные места (цена = 0) проходят фильтр при любом бюджете.

**3. Стиль поездки → мягкий приоритет внутри подборки**

Стиль не убирает места, но поднимает наверх категории, наиболее ему соответствующие (рейтинговый буст +0.5):

| Стиль | Приоритетные категории |
|---|---|
| Культура и история | Музеи, Экскурсии |
| Вечеринки и ночная жизнь | Музыка и бары |
| Природа и парки | Экскурсии, Экстрим |
| Гастрономический тур | Еда |
| Приключения | Экстрим, Экскурсии |
| Всего понемногу | без приоритетов, только по рейтингу |

**4. Темп + Количество дней → лимит выдачи**

| Темп | Мест в день | Пример: 3 дня |
|---|---|---|
| Спокойный | 4 | 12 мест |
| Умеренный | 6 | 18 мест |
| Активный | 10 | 30 мест |

Итоговая логика пайплайна:

```
все места
  → [интересы] оставить только нужные категории
  → [бюджет]   убрать дороже лимита
  → [стиль]    отсортировать: приоритетные категории идут первыми
  → [темп×дни] обрезать до нужного количества
```

## User Flows

### Flow 1 — Новый пользователь, полный сценарий

```
Открывает приложение
  → Видит hero-секцию с поиском
  → Нажимает «Показать» → появляется OnboardingModal
  → Выбирает «Настроить подборку» → OnboardingView (5 шагов)
    [1] Выбирает интересы: музеи, еда
    [2] Выбирает стили: культурный + гастро
    [3] Выбирает бюджет: средний
    [4] Выбирает темп: умеренный
    [5] Выбирает 3 дня → «Начать планирование»
  → PlacesView: персональная подборка загружена
  → Листает карточки, открывает детали мест
  → Добавляет 7-8 мест в подборку
  → Нажимает кнопку корзины → CartView
  → «Составить маршрут» → алгоритм генерирует план
  → PlannerView: 3 дня, переставляет места внутри дней
  → «Сохранить» → SummaryView
  → Добавляет заметки, скачивает .txt, делится маршрутом
```

### Flow 2 — Пользователь пропускает онбординг

```
Открывает приложение
  → Нажимает «Показать» → OnboardingModal
  → Выбирает «Не сейчас»
  → PlacesView: все места без персонализации
  → Фильтрует по категории «Музеи»
  → Открывает панель фильтров, ставит рейтинг ≥ 4.0
  → Добавляет понравившиеся места в подборку
  → Строит маршрут
  → В футере нажимает «Настроить подборку» → проходит онбординг
  → Возвращается в каталог — подборка персонализирована
```

### Flow 3 — Возвращающийся пользователь

```
Открывает приложение (данные из localStorage)
  → Предпочтения сохранены, онбординг не показывается
  → Нажимает Cmd+K → открывается глобальный поиск
  → Вводит «Эрмитаж» → видит результат, нажимает Enter
  → PlaceDetailView: открывает детали, добавляет в избранное ❤️
  → Переходит в «Избранное» через шапку
  → Из избранного добавляет место в подборку
  → Идёт в корзину → строит маршрут
```

### Flow 4 — Редактирование маршрута

```
PlannerView: маршрут уже построен
  → Видит день 1: 4 места, день 2: 3 места
  → Нажимает «✕» рядом с местом — удаляет его
  → Стрелками меняет порядок мест в дне
  → Переносит место из дня 1 в день 2
  → «Сохранить →» → SummaryView
  → Видит обновлённую стоимость и суммарное время
  → Добавляет заметку: «взять зонт, купить билеты заранее»
  → Нажимает «🖨️ Печать»
```

## Технологии

### Vue 3

Vue 3 — прогрессивный JavaScript-фреймворк для построения пользовательских интерфейсов. В проекте используется исключительно Composition API с синтаксисом `<script setup>`: вся реактивность строится через `ref`, `computed`, `watch`; логика выносится в composables (`useSort`, `useFilters`, `useTheme` и др.). Компонентная архитектура позволила чётко разделить UI на переиспользуемые части: карточки мест, модальные окна, секции каталога, блоки планировщика.

По сравнению с **React**: Vue предлагает двустороннее связывание данных (`v-model`) и встроенные директивы (`v-for`, `v-if`), что сокращает количество шаблонного кода. По сравнению с **Angular**: Vue значительно легче в освоении и не требует TypeScript-декораторов и DI-контейнера. Vue был выбран как фреймворк с низким порогом входа, развитой экосистемой и хорошей документацией на русском языке.

### TypeScript

TypeScript — строго типизированный надстройка над JavaScript. В проекте включён режим `strict`: все сущности (типы `Place`, `UserPreferences`, `DayPlan`, `Category`), store, composables и props компонентов полностью типизированы. Это позволило поймать ряд ошибок на этапе разработки — например, несовпадение типа `travelStyle` после изменения с `string` на `string[]`.

По сравнению с чистым **JavaScript**: TypeScript требует больше начальных усилий, но при росте кодовой базы многократно окупается: IDE подсказывает доступные поля, рефакторинг безопаснее, ошибки типов не уходят в рантайм. Альтернативой мог быть **JSDoc** для аннотаций без компиляции, но он менее удобен для сложных обобщённых типов.

### Vite

Vite — современный инструмент сборки фронтенда, использующий нативные ES-модули браузера в режиме разработки. Dev-сервер стартует мгновенно независимо от размера проекта, а Hot Module Replacement (HMR) обновляет конкретный компонент без перезагрузки страницы. В проекте Vite также настраивает path alias `@/` для удобных импортов.

По сравнению с **webpack**: Vite в 10–100 раз быстрее при холодном старте и горячей замене модулей, так как не бандлит весь код перед запуском. По сравнению с **Parcel**: Vite лучше интегрируется с экосистемой Vue и предлагает более гибкую конфигурацию через `vite.config.ts`. Выбор Vite очевиден для новых Vue-проектов — он является официально рекомендуемым инструментом.

### Vue Router 4

Vue Router — официальная библиотека маршрутизации для Vue. В проекте реализована SPA-навигация между восемью страницами: главная, онбординг, каталог, детали места, корзина, планировщик, итог, избранное. Используются именованные маршруты, query-параметры (например, `/places?search=эрмитаж&category=museums`) и программная навигация через `useRouter()`.

По сравнению с **React Router**: Vue Router теснее интегрирован с фреймворком, а navigation guards проще в написании. Альтернативой мог быть **unplugin-vue-router** с файловой маршрутизацией, но для учебного проекта явная конфигурация маршрутов нагляднее.

### Pinia

Pinia — официальное решение для управления глобальным состоянием в Vue 3. В проекте используется пять store: `userStore` (предпочтения и онбординг), `placesStore` (места и рекомендации), `cartStore` (подборка мест), `plannerStore` (маршрут по дням), `favoritesStore` (избранное). Каждый store использует setup-стиль — те же `ref` и `computed`, что и в компонентах, без лишней абстракции. Состояние частично персистируется в `localStorage`.

По сравнению с **Vuex** (предыдущий стандарт Vue): Pinia не требует мутаций, модулей и пространства имён — код значительно лаконичнее. По сравнению с глобальными composables: Pinia предоставляет инструменты devtools, поддержку плагинов и более предсказуемую персистентность.

### MSW (Mock Service Worker)

MSW перехватывает fetch-запросы на уровне Service Worker в браузере и возвращает заранее описанные ответы — без запуска реального сервера. В проекте реализованы пять эндпоинтов: получение мест, детали места, рекомендации по предпочтениям, генерация маршрута и его пересчёт. Ключевое преимущество — одни и те же handlers используются как в браузере, так и в тестах Vitest, что исключает дублирование фикстур.

По сравнению с **json-server**: MSW работает прямо в браузере без отдельного процесса, поддерживает сложную логику в обработчиках и интегрируется с тестами. По сравнению с ручными моками в тестах (`vi.mock`): MSW перехватывает реальные HTTP-запросы, поэтому тесты проверяют полный путь от fetch-вызова до обработки ответа.

### Vitest и @vue/test-utils

Vitest — тестовый фреймворк, встроенный в экосистему Vite. Использует тот же конфиг и те же трансформации, что и основной проект, поэтому не нужно отдельно настраивать TypeScript, алиасы и обработку `.vue`-файлов. В проекте написано 174 теста в 19 файлах: покрыты store, composables, алгоритм планировщика, компоненты и MSW handlers.

По сравнению с **Jest**: Vitest быстрее (параллельное выполнение, нативные ESM), не требует babel-трансформаций и работает из коробки с Vite-проектом. `@vue/test-utils` — официальная библиотека для монтирования Vue-компонентов в тестах, аналог **React Testing Library** для Vue.

### CSS Custom Properties

CSS-переменные (Custom Properties) — нативный механизм браузера для хранения значений, используемых в стилях. В проекте реализована полноценная дизайн-система: переменные для цветов, типографики, отступов, теней и радиусов объявлены в `:root`, а тёмная тема переопределяет только нужные переменные через `[data-theme="dark"]`. Тема применяется мгновенно без перерисовки компонентов.

По сравнению с **CSS-in-JS** (styled-components, emotion): CSS-переменные не требуют зависимостей и работают на уровне браузера без рантайм-стоимости. По сравнению с **SCSS-переменными**: CSS-переменные живут в DOM и меняются динамически через JavaScript, что делает их единственным нативным способом реализовать runtime-переключение темы.

## Заключение

Composition API Vue 3 оказался удобнее, чем ожидалось: логика читается линейно, composables легко переиспользуются. Pinia поначалу казалась избыточной, но когда состояние корзины понадобилось в пяти разных местах — всё встало на место.
MSW приятно удивил: идея перехватывать fetch на уровне Service Worker настолько элегантна, что сложно представить проект без реального API, где стоит  использовать что-то другое. Особенно ценно, что тесты и браузер работают с одними handlers.    
Главной проблемой оказалось то, что данные мест не персистировались: при перезагрузке планировщик показывал пустые карточки. Пришлось написать `usePlacesInit`, который проверяет и при необходимости подгружает данные на любой странице.
